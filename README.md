# Neoflex Meeting Transcriber

## Проблема

При проведении рабочих встреч возникает необходимость получать протокол разговора с указанием говорящих. Современные speech-to-text модели способны выполнять транскрибацию аудио, однако определение личности говорящего (speaker identification) остается отдельной задачей.

Существующие решения ограничиваются **диаризацией**, не сопоставляя участников встречи с реальными людьми организации.

Данный проект должен решить эту проблему за счет хранения базы голосовых семплов сотрудников и последующей идентификации говорящих при обработке новых встреч.

Дополнительно система должна позволять постепенно улучшать качество распознавания за счет подтверждения результатов человеком и безопасного накопления голосовых эмбеддингов.

---

## Цель

Разработать сервис фоновой обработки записей встреч, который:

* выполняет автоматическую транскрибацию аудио;
* разделяет речь различных участников;
* идентифицирует говорящих по ранее накопленным голосовым семплам;
* формирует итоговый протокол встречи;
* позволяет организатору подтверждать либо исправлять идентификацию участников;
* постепенно пополняет базу голосовых эмбеддингов;
* обеспечивает защиту базы голосовых отпечатков от повторного использования.

---

## Функциональные требования

На вход сервис получает:

* аудиофайл встречи (или Base64-представление);
* e-mail организатора встречи.

На выходе сервис возвращает:

* текстовую расшифровку встречи;
* список участников;
* имя говорящего для каждой реплики (если удалось идентифицировать);
* пометку `Unknown` для неизвестных участников.

---

## Конвеер обработки

```text
Meeting Record
        ↓
Silero VAD
(поиск участков речи)
        ↓
pyannote.audio
(диаризация)
        ↓
Speechbrain (ECAPA-TDNN)
(получение embedding)
        ↓
Embedding (192d)
        ↓
Qdrant Vector DB
        ↓
Top-K Search
        ↓
Similarity Threshold
        ↓
Имя человека / Unknown
```

---

## Алгоритм идентификации говорящего

Для каждого сегмента речи:

1. Получить embedding при помощи Speechbrain (ECAPA-TDNN).
2. Выполнить поиск Top-10 ближайших centroid в Qdrant.
3. Для каждого найденного кандидата:
   * сравнить embedding сегмента со всеми embeddings данного человека;
   * вычислить максимальную cosine similarity.
4. Выбрать лучший результат.
5. Если similarity превышает установленный порог:
   * вернуть имя человека.
6. Иначе:
   * вернуть `Unknown`.

## Защита голосовых семплов

Для защиты предлагается использовать следующие меры:

### Шифрование

Все embeddings хранятся только в зашифрованном виде (AES-256).

---

### Cекретная матрица преобразования

Перед сохранением, embedding преобразуется с помощью секретной обратимой матрицы:

$$
e' = M \cdot e
$$

где:

* **e** — исходный embedding, полученный моделью ECAPA-TDNN;
* **M** — секретная матрица преобразования;
* **e'** — преобразованный embedding, который сохраняется в базе данных.

При идентификации нового пользователя его embedding проходит то же преобразование перед поиском ближайших соседей в векторной базе.

Матрица преобразования ортогональная для сохранения взаимного расположения векторов в пространстве.

## Дополнительные агенты

### Агент пополнения базы голосов

Назначение:

* анализировать неизвестных говорящих;
* показывать организатору встречи предполагаемые совпадения;
* подтверждать либо исправлять идентификацию;
* автоматически добавлять новые embeddings в профиль пользователя после подтверждения.

---

### Агент контроля качества протокола

Назначение:

* получать оценку итогового протокола от организатора;
* анализировать ошибки;
* определять причины низкого качества:
  * плохая диаризация;
  * ошибки STT;
  * ошибки идентификации;
* формировать рекомендации по улучшению модели;
* инициировать повторное обучение базы голосов.

## Use Cases

### UC-1. Автоматическая обработка встречи

1. Пользователь загружает запись встречи.
2. Сервис выполняет транскрибацию.
3. Выполняется диаризация.
4. Идентифицируются говорящие.
5. Формируется протокол.

---

### UC-2. Подтверждение участников

1. Организатор получает список неизвестных говорящих.
2. Подтверждает личности.
3. Система сохраняет новые embeddings.
4. Качество идентификации улучшается.

---

### UC-3. Повторная встреча

1. Загружается новая запись.
2. Система использует накопленные embeddings.
3. Большинство участников определяется автоматически.

---

### UC-4. Оценка качества

1. Организатор оценивает качество протокола.
2. Агент собирает обратную связь.
3. Формируются рекомендации по улучшению качества распознавания.
